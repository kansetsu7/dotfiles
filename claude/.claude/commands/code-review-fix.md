Execute fixes from the fix plan, working through each group in order.

## Prerequisites

- `.claude/fix-plan.md` must exist (generated by `/fix-plan`)
- `.claude/code-review.md` must exist with human decisions filled in
- All "Needs Decision" and "Design Decisions Needed" items should be resolved before running

## Workflow

### Step 1: Read and Validate Plan

1. Read `.claude/fix-plan.md`
2. Read `.claude/code-review.md`
3. Check for unresolved blockers:
   - If "Needs Decision" items remain → list them and ask user to resolve first
   - If "Design Decisions Needed" items have no chosen option → list them and ask user to decide
4. Build the work queue from Fix Groups, respecting group order

### Step 2: Process Each Group

For each Fix Group in order:

1. **Announce the group:**
   ```
   ## Group N: <theme>
   ```

2. **For each item in the group** where Status is `[ ]` (unchecked) in `code-review.md`:

   a. **Check Approach:**
      - If "Suggested fix" → implement the Suggestion from `code-review.md`
      - If "Alternative" → implement the approach from Notes in `code-review.md`
      - If the fix-plan has a specific Plan description → follow that

   b. **Check Notes for questions:**
      - If Notes contain questions (indicated by `?`) → answer the question first, explain reasoning, then proceed

   c. **Implement and commit:**
      - Follow the **Todo** checklist in `fix-plan.md`, completing each sub-task in order
      - Mark each todo as `[x]` in `fix-plan.md` as you go
      - Verify the change compiles/passes syntax check
      - Mark the item Status as `[x]` in `.claude/code-review.md`
      - Stage the changed files (including updated `.claude/code-review.md` and `.claude/fix-plan.md`)
      - Commit with a message describing the changes made

   d. **Check cascading fixes:**
      - If fix-plan notes this item resolves others, mark those as `[x]` in `code-review.md` with note "Resolved by #X"

   e. **Report:**
      ```
      Fixed: `<file_path:line>` - <brief title>
      Approach: <Suggested fix | Alternative: brief description>
      ```

### Step 3: Skip Non-Accepted Items

- **Drop**: Skip silently
- **Won't Fix**: Skip silently
- Already checked `[x]`: Skip (already fixed)

### Step 4: Summary Report

After processing all groups, provide a summary:

```markdown
## Code Review Fix Summary

### Fixed (<count>)
- 🤖 AI: <count> items
- 👤 Reviewer: <count> items
- 🤖 AI + 👤 Reviewer: <count> items

**Details:**
- `<file_path:line>` - <title> (Suggested fix) [🤖 AI]
- `<file_path:line>` - <title> (Alternative) [👤 Reviewer]

### Resolved by Cascading Fix (<count>)
- `<file_path:line>` - <title> (resolved by #X) [<source>]

### Skipped - Drop (<count>)
- `<file_path:line>` - <title> [<source>]

### Skipped - Won't Fix (<count>)
- `<file_path:line>` - <title> [<source>]

### Already Done (<count>)
- `<file_path:line>` - <title> [<source>]

### Remaining (<count>)
- `<file_path:line>` - <title> (no decision yet) [<source>]
```

## Important Notes

- **Follow group order** from fix-plan.md — groups encode dependency ordering
- **Conflicts**: If fix-plan flagged conflicts, follow the chosen resolution
- **Cascading fixes**: After each fix, check fix-plan for cascading resolutions
- If an alternative approach is unclear, ask for clarification before implementing
- Run project tests after all fixes if a test command is available
